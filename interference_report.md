# Twitch Extension Interference Report

## 1. Executive Summary
Currently, the stream overlay Chrome extension works smoothly on standard Twitch streams. However, when a viewer encounters a stream with active Twitch Extensions (like the Destiny 2 Armory, Sound Alerts, etc.), the overlay stops capturing or triggering correctly.

This research report isolates the root cause to **Cross-Origin iframe Event Swallowing** and outlines an architectural workaround that allows our content script to track the mouse position globally without breaking existing Twitch Extensions.

---

## 2. The Core Problem
Twitch Extensions are rendered as transparent `<iframe>` elements that sit directly on top of the main Twitch video player. These iframes are hosted on a different domain (`ext.twitch.tv`).

In the current architecture, [content.js](file:///c:/ClaudeCodeProjects/twitch-overlay/extension/content.js) relies on listening to `document.addEventListener("mousemove")` in the top-level main DOM. When the user's mouse pointer enters the bounding box of a Twitch Extension iframe:
1. The mouse events are captured by the `ext.twitch.tv` iframe's document.
2. Cross-origin security policies (CORS) prevent these `mousemove` events from bubbling up to the parent `twitch.tv` window.
3. Our [content.js](file:///c:/ClaudeCodeProjects/twitch-overlay/extension/content.js) loses track of the mouse (`lastMouseX` / `lastMouseY` stop updating), which freezes the capture loop and prevents hash matching until the mouse leaves the iframe.

Additionally, [findBestVideo()](file:///c:/ClaudeCodeProjects/twitch-overlay/extension/content.js#66-81) queries all `<video>` tags. If a Twitch Extension uses a background or hidden `<video>` element internally, it can occasionally steal focus from the actual Twitch player if it's the only one detected dynamically.

---

## 3. Recommended Solution: All-Frames Injection & postMessage

Because we cannot force `pointer-events: none` onto Twitch Extensions without breaking their own interactivity, we must instead inject our tracking script directly inside them and relay the cursor position back up to the main extension script.

### Step 3.1: Update [manifest.json](file:///c:/ClaudeCodeProjects/twitch-overlay/extension/manifest.json)
Update the `content_scripts` to run inside child iframes. `*.twitch.tv` legitimately matches `ext.twitch.tv`.

```json
  "content_scripts": [
    {
      "matches": ["https://*.twitch.tv/*"],
      "js": ["content.js"],
      "run_at": "document_idle",
      "all_frames": true 
    }
  ]
```

### Step 3.2: Frame Detection and Global Relay
Refactor [content.js](file:///c:/ClaudeCodeProjects/twitch-overlay/extension/content.js) to behave differently depending on whether it has spawned in the "top" frame (the main Twitch page) or a child iframe (the Twitch Extension).

**In the Child Frame (`window !== window.top`):**
Instead of attempting a video capture (which it cannot do because the main `<video>` is securely in the parent frame), it should strictly serve as a mouse relay. 
```javascript
if (window !== window.top) {
    document.addEventListener("mousemove", (e) => {
        // Send internal coordinates up to the main twitch tab
        window.parent.postMessage({
            type: "streamGenie_mousemove",
            clientX: e.clientX,
            clientY: e.clientY
        }, "*"); // Securely route to the parent twitch frame
    }, { passive: true });
}
```

**In the Top Frame (`window === window.top`):**
The top frame listens for these messages, identifies exactly which iframe sent the message to compute its global physical offset on the screen, and logically processes the capture as if the mouse event had fired locally.

```javascript
window.addEventListener("message", (event) => {
    if (event.data && event.data.type === "streamGenie_mousemove") {
        // Find which iframe this secure message came from
        const iframes = Array.from(document.querySelectorAll("iframe"));
        const sourceFrame = iframes.find(f => f.contentWindow === event.source);
        
        if (sourceFrame) {
            const rect = sourceFrame.getBoundingClientRect();
            
            // Translate iframe coordinates into absolute top-level coordinates
            const syntheticClientX = rect.left + event.data.clientX;
            const syntheticClientY = rect.top + event.data.clientY;
            
            // Fire our existing logic using the calculated position!
            processVideoCapture(syntheticClientX, syntheticClientY);
        }
    }
});
```

## 4. Secondary Mitigation: Strict Video Player Selection
To prevent [findBestVideo()](file:///c:/ClaudeCodeProjects/twitch-overlay/extension/content.js#66-81) from incorrectly latching onto videos generated by Twitch Extensions (which are now accessible if `"all_frames": true` is enabled and might confuse standard loop executions), constrain video detection to specific selector pathways instead of sweeping all sizes.

**Mitigation:** 
Check that the target video is a child of Twitch's explicit player container:
```javascript
function findBestVideo() {
    const all = Array.from(document.querySelectorAll(".video-player__container video, .persistent-player video"));
    //... continue existing sorting logic
}
```

## 5. Conclusion
By utilizing `"all_frames": true` combined with an `event.source` mapping via `window.postMessage`, the extension can entirely bypass the cross-origin pointing event blackout created by third-party Twitch Extensions, retaining a seamless overlay experience for the user.
