Add shallow schema validation in applyProfile() (ERR-03) and wire the error flags from
Plan 01's extended get-game response into the popup's profile-selector note element
(ERR-01/ERR-02 reflected in popup per D-03).

or runtime errors. Invalid triggers should be skipped with a visible diagnostic. Additionally,
the popup currently has no awareness of content.js load errors — the user sees "no profile"

schema-skip debug panel line. popup.js with get-game response reader that sets the existing
note element to reflect profileLoadError (red) or profileStaleWarning (amber).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| profile.json -> applyProfile() | Untrusted profile data from CDN enters validation loop |
| content.js -> popup.js via get-game | Error strings from content.js appear in popup DOM |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-03 | Tampering | profile.json trigger fields used in profileSchemaWarnings | accept | trigger.id values from CDN profile are developer-authored identifiers, not viewer-supplied. Shown only in debug panel and console. Low risk. |
| T-12-04 | Information Disclosure | resp.profileLoadError rendered in popup via textContent | mitigate | textContent used (not innerHTML) — XSS not possible. Error string is a JS runtime message from fetch (e.g. "HTTP 404", "Failed to fetch"), not profile content. |
| T-12-05 | Information Disclosure | resp.profileStaleWarning rendered in popup | mitigate | Same — textContent only, value is err.message from a fetch Error object. No profile-supplied content reaches the DOM. |
</threat_model>
